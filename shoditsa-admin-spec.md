## 1. Назначение

Создать закрытую административную панель для ежедневного управления проектом «Сходится!». Панель должна позволять одному администратору:

1. Быстро находить, проверять и редактировать игровые карточки во всех существующих категориях.
2. Получать и разбирать пользовательские сообщения об ошибках, сохраняя связь отчёта с карточкой, раундом и версией данных.
3. Запускать существующий ИИ-пайплайн обработки музыки, видеть результат по каждому полю и применять одобренные изменения.
4. Просматривать пользователей и подробную хронологию их действий.
5. Контролировать качество контента, фоновые задачи и собственные административные действия.

Админка предназначена для desktop-first работы. Основная поддерживаемая ширина — от 1280 px. На планшете от 1024 px должны работать просмотр, поиск и простые действия. Отдельная мобильная версия в первый релиз не входит.

## 2. Зафиксированная архитектура проекта

ТЗ составлено по фактическому состоянию `main` указанного выше commit.

| Слой | Текущее решение | Требование к админке |
| --- | --- | --- |
| Frontend | React 19 + Vite 8 + TypeScript, `apps/web` | Админка создаётся внутри `apps/web`, отдельным lazy-loaded разделом `/admin/*` |
| Server state | TanStack Query 5 | Списки, мутации и обновление прогресса используют общий query client |
| Backend | Fastify 5, модульный монолит, `apps/api` | Новые маршруты выносятся из растущего `app.ts` в `apps/api/src/modules/admin/*` |
| Контракты | TypeBox + OpenAPI, `packages/contracts` | Каждый запрос и ответ админки получает runtime-схему и TypeScript-тип |
| БД | PostgreSQL 18 + Drizzle ORM, `packages/database/src/schema.ts` | Новые сущности добавляются forward-only миграциями |
| Авторизация | Better Auth, anonymous plugin, email/Yandex | Используется текущая cookie-сессия; admin guard усиливается согласно разделу 4 |
| Контент | `content_items`, immutable `content_item_versions`, цельные `content_revisions` | Редактор создаёт рабочие изменения и выпускает новую цельную ревизию; активные версии не обновляются на месте |
| Игры | `game_sessions`, `game_attempts`, `game_hint_choices` | Подробная timeline строится прежде всего по существующим таблицам |

## 4. Доступ и безопасность

### 4.1. Идентификация администратора

В репозитории уже существуют `player_profiles.role`, `requireAdmin()` и конфигурация `ADMIN_EMAILS`, однако текущий код `session.ts` проверяет только роль профиля, а `ADMIN_EMAILS` саму роль не назначает. Это необходимо исправить явно.

1. Добавить одноразовую команду `npm run admin:bootstrap -- --email=breneize@yandex.ru`.
2. Команда нормализует email, находит существующую строку `user`, завершается ошибкой при нуле или нескольких совпадениях, выводит найденный UUID и обновляет/создаёт `player_profiles` с `role = 'admin'`.
3. Стабильный UUID записывается в production secret/config `ADMIN_USER_IDS`. Email остаётся в `ADMIN_EMAILS` как понятный владельцу bootstrap allowlist.
4. `requireAdmin()` разрешает доступ при одновременном выполнении условий: валидная Better Auth session, `player_profiles.role = 'admin'`, user ID входит в `ADMIN_USER_IDS`, email после нормализации совпадает с `breneize@yandex.ru`.
5. Startup validation в production требует ровно один `ADMIN_USER_IDS` и ровно один `ADMIN_EMAILS`; лишние значения блокируют запуск API до исправления конфигурации.
6. Интерфейс не содержит назначения ролей. API не предоставляет endpoint выдачи admin role.
7. Все остальные пользователи получают `403 ADMIN_REQUIRED` от каждого `/api/v1/admin/*`.

Email выступает дополнительным инвариантом. Основная стабильная идентичность — UUID существующего пользователя.

### 4.2. Обязательные меры

- Проверка доступа выполняется на сервере для каждого административного API и серверного действия.
- Ссылка на админку отображается только разрешённому пользователю.
- Клиентское скрытие элементов считается дополнительной мерой интерфейса.
- Для cookie-сессий обязательны `HttpOnly`, `Secure`, подходящий `SameSite` и CSRF-защита для изменяющих запросов.
- Для опасных действий требуется повторное подтверждение в модальном окне.
- Секреты пайплайна, токены и пароли никогда не возвращаются в браузер и не попадают в логи.
- Административные ответы используют заголовок `Cache-Control: no-store`.
- Доступ к изображениям и экспортам выдаётся через контролируемые URL с ограниченным сроком жизни, если файлы не публичные.
- При блокировке пользователя существующие активные сессии должны быть инвалидированы, если это поддерживает текущая система авторизации.
- Администратор не может заблокировать, удалить, анонимизировать или отозвать все сессии у самого себя; запрет действует на сервере.
- `/api/v1/metrics` с bearer token остаётся отдельным техническим endpoint и не заменяет admin guard.

## 7. Карточки и работа с контентом

### 7.1. Общая модель интерфейса

Основной рабочий экран строится как таблица с открывающейся справа широкой панелью карточки. Администратор сохраняет контекст списка и быстро переходит к следующей записи.

Режимы:

- **Таблица** — массовая работа и поиск;
- **Карточки** — визуальный просмотр сеткой;
- **Проверка** — последовательный просмотр записей, попавших под выбранный фильтр.

Состояние фильтров, сортировки и выбранных колонок сохраняется для администратора.

Над списком постоянно видна строка рабочей версии:

- базовая active revision;
- количество несохранённых и сохранённых draft-изменений;
- число ошибок и предупреждений валидации;
- кнопка «Проверить изменения»;
- кнопка «Опубликовать N изменений».

В MVP одновременно существует одна открытая рабочая версия, созданная от текущей active revision. Она хранится на сервере и доступна после смены устройства или браузера.

Универсальный редактор произвольных PostgreSQL-таблиц и выполнение SQL из браузера в MVP не входят. Работа с БД представлена предметными экранами карточек, пользователей, репортов, экономики, ревизий и логов; это сохраняет constraints и audit каждой операции.

### 7.2. Таблица карточек

Базовые колонки:

- чекбокс;
- миниатюра;
- основное название;
- внутренний ID;
- категория/режим;
- статус публикации;
- язык;
- индикатор полноты данных;
- число открытых баг-репортов;
- предупреждения качества;
- дата последнего изменения;
- источник последнего изменения: вручную / импорт / ИИ;
- меню действий.

Фильтры:

- категория/режим;
- опубликована / скрыта / архивна;
- язык;
- есть открытые баг-репорты;
- нет изображения / битое изображение;
- отсутствуют обязательные поля;
- возможный дубликат;
- изменена за период;
- источник изменения;
- статус последнего ИИ-прогона;
- пользовательские сохранённые наборы фильтров.

Поиск должен поддерживать:

- название и его часть;
- альтернативное и оригинальное название;
- внутренний ID;
- внешний ID, если он есть;
- точное совпадение в кавычках;
- поиск без учёта регистра и с нормализацией `ё/е`.

Сортировка: название, ID, дата создания, дата обновления, количество репортов, полнота.

Кнопка **«Добавить карточку»** открывает выбор режима и пустую форму его схемы. ID создаётся по действующим правилам режима либо вводится вручную в расширенном блоке, затем проверяется на уникальность глобально в `content_items` и в workspace. После первого сохранения ID и mode становятся неизменяемыми. Физическое удаление контентных identity из UI отсутствует; карточка исключается через `allowedInGame = false` и новую revision.

### 7.3. Массовые действия

- добавить выбранные изменения в рабочую версию;
- разрешить/запретить участие в игре через `allowedInGame`;
- добавить/удалить тег;
- назначить значение поддерживаемого справочного поля;
- экспортировать выбранное в JSON/CSV;
- добавить в запуск музыкального пайплайна, если все выбранные карточки относятся к музыке;
- запустить проверку качества;
- закрыть связанные однотипные предупреждения после проверки.

Перед применением показывается число затронутых записей и краткое описание эффекта. Массовые изменения сначала попадают в рабочую версию. Для крупных выборок они выполняются фоновой задачей. Каждая логическая операция фиксируется в `audit_log`.

### 7.4. Редактор карточки

Панель редактора занимает 60–70% ширины окна и содержит:

1. Заголовок, ID, статус и быстрые действия.
2. Вкладку **Данные**.
3. Вкладку **Как в игре**.
4. Вкладку **Баг-репорты**.
5. Вкладку **История**.
6. Вкладку **Техническое**.

#### Вкладка «Данные»

Поля группируются по смыслу, например:

- идентификация и названия;
- игровые данные и подсказки;
- допустимые ответы и алиасы;
- фактические метаданные;
- медиа;
- локализации;
- источник и служебные данные;
- публикация.

Форма строится по схеме конкретной категории. Типы контролов соответствуют типам данных: текст, многострочный текст, число, дата/год, enum, теги, массив алиасов, URL, изображение, JSON только для технических полей.

Состав формы опирается на текущий `TitleItem` из `packages/contracts/src/legacy-types.ts`:

| Режим | Специализированные группы полей |
| --- | --- |
| Все | `id` и `mode` только для чтения; `titleRu`, `titleOriginal`, `alternativeTitles`, годы/дата, страны, язык, жанры, popularity/top rank, описания, `plotHint`, slogan, facts, notes, media, `allowedInGame` |
| Кино | runtime, age rating, budget, directors, writers, cast/supporting cast, Kinopoisk/IMDb IDs, ratings/votes, awards |
| Сериалы | episodes, seasons count, series status, showrunners, writers, cast, ratings/votes, Kinopoisk/IMDb IDs |
| Аниме | kind/status/episodes aired/source, studios, directors/cast, Shikimori ID/score/URL, ratings |
| Игры | developers, publishers, platforms, Steam categories/tags/languages, Steam app ID/URL, price, Metacritic и игровые votes/recognizability |
| Музыка | canonical ID, aliases, game tier, content status, active state, origin, artist type, top tracks/albums, similar artists, members/associated acts под предметными подписями, music links, data quality |
| Диагнозы | ICD-10/group, body systems, disease types, course, contagiousness, age/sex, localization, symptoms, diagnostics, risks, severity/urgency, safety disclaimer и связанные case vignettes |

Поля, отсутствующие в конкретном payload, отображаются пустыми только если они разрешены схемой режима. Компонент формы не должен выдавать весь `TitleItem` как один одинаковый длинный список.

Для каждого поля доступны:

- понятная подпись и короткая подсказка;
- обязательность;
- текущее значение;
- индикатор ошибки или предупреждения;
- источник значения, если он известен;
- кнопка просмотра предыдущего значения;
- счётчик символов для полей с лимитом.

Режим **«Проверка»** переиспользует текущие `payload.reviewReasons` и `content_review_decisions`. Решение по полю остаётся в существующей таблице, а действие «Применить решение» создаёт соответствующий `content_workspace_change`. Служебное решение `field = '__approval__'` завершает проверку карточки, но само по себе не публикует контент.

Поведение сохранения:

- несохранённые изменения явно отмечаются;
- `Ctrl/Cmd + S` сохраняет карточку;
- при попытке закрыть карточку с изменениями появляется подтверждение;
- перед записью выполняется клиентская и серверная валидация;
- успешное сохранение записывает полную предложенную payload карточки в текущую `content_workspace`, не изменяя active `content_item_versions`;
- сохранённое изменение содержит base item version, автора, источник, diff и номер версии;
- повторное открытие карточки показывает active-значение и поверх него сохранённый draft;
- конкурентное изменение определяется по `workspace_change.version`; при конфликте показывается сравнение;
- рядом доступны действия «Отменить правку» и «Вернуть к active-версии»;
- браузер локально сохраняет временный несохранённый текст при случайной перезагрузке.

Кнопка **«Сохранить и опубликовать»** разрешена для одной карточки: она сохраняет draft, запускает сборку новой цельной content revision, показывает итог проверки и активирует её после подтверждения. Технически операция использует тот же workflow, что массовая публикация.

#### Вкладка «Как в игре»

- Используется реальный компонент игровой карточки или максимально общий с ним компонент из той же кодовой базы.
- Доступны размеры desktop и mobile.
- Можно последовательно раскрыть подсказки и проверить допустимые ответы.
- Предпросмотр обновляется от несохранённых данных формы.
- Показывается предупреждение, если реальный вид карточки невозможно воспроизвести из-за отсутствующего медиа.

#### Вкладка «Баг-репорты»

- Список всех отчётов по карточке, новые сверху.
- Причина, текст, дата, пользователь, контекст раунда и статус.
- Быстрые действия «В работу», «Исправлено», «Отклонить», «Дубликат».
- После сохранения исправления можно связать ревизию с одним или несколькими отчётами и закрыть их одним действием.

#### Вкладка «История»

- хронология значений карточки по `content_item_versions` и записей рабочей версии;
- отдельная отметка active revision;
- автор/процесс, дата, причина, связанные репорты и pipeline run;
- diff по полям;
- действие «Взять это значение в рабочую версию» с предварительным diff;
- восстановление никогда не обновляет историческую item version.

#### Вкладка «Техническое»

- JSON текущей карточки с подсветкой синтаксиса;
- копирование JSON;
- внешние ID и ссылки на источники;
- технические даты и версия схемы;
- редактирование JSON доступно только через режим «Расширенное редактирование», с серверной проверкой схемы и отдельным подтверждением.

### 7.5. Медиа

- загрузка изображения перетаскиванием или выбором файла;
- предпросмотр итогового кропа;
- проверка формата, размера и минимального разрешения;
- замена, удаление, копирование URL;
- выявление недоступных ссылок;
- старая ссылка сохраняется в истории ревизии;
- SVG санитизируется либо запрещается согласно текущей политике проекта.

### 7.6. Сборка и публикация рабочей версии

Текущие `content_revisions` представляют полный immutable snapshot всех шести библиотек. Админка сохраняет этот инвариант.

Алгоритм публикации:

1. Заблокировать workspace от новых изменений и зафиксировать его базовую active revision.
2. Убедиться, что active revision не сменилась с момента создания workspace. При смене выполнить rebase draft-изменений и показать конфликты по конкретным карточкам.
3. Взять полный набор `content_item_versions` базовой ревизии.
4. Применить draft payload к изменённым карточкам, добавить новые, исключить только явно удалённые/запрещённые записи.
5. Повторно вычислить типизированные колонки `titleRu`, `titleOriginal`, `normalizedTitle`, `year`, `popularityScore`, `sortOrder`, `allowedInGame`, aliases и diagnosis vignettes из итоговых payload.
6. Запустить общую валидацию, извлечённую из `scripts/content/lib.ts`/текущего importer в переиспользуемый сервис. CLI и admin publish должны использовать одну реализацию правил.
7. Проверить counts по шести режимам, checksum и защиту от падения количества более 5%.
8. В одной контролируемой операции создать новую `content_revisions` со статусом `importing`, все item versions/aliases/vignettes и перевести её в `ready`.
9. Показать администратору итог: изменённые карточки, counts, warnings, checksum и влияние на игру.
10. После подтверждения активировать готовую revision через общий сервис, который переиспользует логику текущего `/api/v1/admin/content/revisions/:id/activate`.
11. Закрыть workspace как `published`, создать пустой workspace от новой active revision и сохранить audit trail.

При ошибке новая revision получает `failed`; active revision остаётся прежней. Частично созданная revision никогда не участвует в игре.

### 7.7. Влияние публикации на игры

- Текущие и завершённые `game_sessions` сохраняют свои `revisionId` и `answerItemVersionId`.
- Уже созданные `daily_challenges` сохраняют закреплённый ответ и не меняются обычной публикацией.
- Новая active revision используется при создании новых challenges и для каталога/free play согласно текущей серверной логике.
- Если будущие challenges materialized на 30 дней вперёд, экран публикации сообщает дату, с которой изменения начнут участвовать в daily.
- Срочная замена уже materialized challenge оформляется отдельным опасным действием «Заменить будущую загадку». В MVP разрешены только даты позже текущей московской даты, отсутствие начатых сессий и обязательный audit. Изменение текущей загадки дня в MVP запрещено.

## 8. Баг-репорты

### 8.1. Данные, которые должен присылать игровой интерфейс

Текущий endpoint `POST /api/v1/content-reports` уже принимает `sessionId`, reason и comment, а server сам записывает `userId`, `itemId` и mode. Этот подход сохраняется.

Payload клиента расширяется только диагностическими полями:

- `clientEventId` для защиты от повторной отправки;
- `appVersion`;
- `pageUrl` без query-параметров с персональными данными;
- `clientErrorId`, если отчёт отправляется из error state.

Остальной контекст сервер восстанавливает по существующим связям:

- `content_reports.sessionId → game_sessions` даёт пользователя, режим, дату, статус, число попыток, rules version и content revision;
- `game_sessions.answerItemVersionId → content_item_versions` даёт неизменяемый снимок ответной карточки на момент игры;
- `game_attempts` даёт все выбранные варианты и response/hints snapshots;
- `game_hint_choices` даёт открытые подсказки;
- новое `game_sessions.auth_session_id` связывает раунд с конкретной Better Auth session и её user agent. Для старых строк без этого поля UI показывает device как «не определено».

Дублировать полный снимок карточки внутри `content_reports` не требуется: существующая immutable item version уже выполняет эту роль.

Типы проблемы:

- `wrong_fact` — неверный факт;
- `disputed_comparison` — спорное сравнение в ответе;
- `title_not_found` — правильный вариант отсутствует в поиске/не принимается;
- `bad_hint` — неверная или слабая подсказка;
- `bad_image` — неверное или битое изображение;
- `duplicate_card` — дубликат карточки;
- `typo_or_translation` — опечатка или плохой перевод;
- `technical_error` — техническая ошибка;
- `other` — другое.

Первые пять текущих значений сохраняются без переименования для совместимости. Новые значения добавляются в TypeBox schema и DB check constraint одной миграцией.

### 8.2. Очередь администратора

Представление — список слева и детальная панель справа. Статусы:

`open → in_progress → resolved`  
`open/in_progress → dismissed`  
`open/in_progress → duplicate`

Существующие значения `open`, `resolved`, `dismissed` сохраняются. Миграция добавляет `in_progress` и `duplicate`.

Фильтры:

- статус;
- причина;
- категория;
- карточка;
- автор;
- дата;
- с комментарием / без комментария;
- несколько отчётов на одну карточку;
- техническая ошибка с correlation ID.

Детальная панель одновременно показывает:

- текст и контекст отчёта;
- item version, закреплённую за сессией;
- карточку из текущей active revision и draft из открытой workspace;
- разницу между этими состояниями;
- все попытки, подсказки, статус и длительность игрового раунда;
- историю похожих отчётов;
- кнопку «Открыть редактор рядом».

При закрытии отчёта требуется выбрать итог:

- исправлено ревизией;
- уже исправлено ранее;
- ожидаемое поведение;
- недостаточно данных;
- дубликат другого отчёта;
- другое с коротким комментарием.

Система хранит автора и дату смены статуса. В будущем на этом основании можно добавить уведомление пользователю; в MVP внешняя отправка уведомления не нужна.

Для этого существующая таблица `content_reports` расширяется полями `updated_at`, `assigned_to`, `resolution_type`, `resolution_comment`, `linked_workspace_change_id`, `linked_revision_id`, `duplicate_of_report_id`, `client_event_id`, `app_version`, `page_url`, `client_error_id`, `request_id`. `client_event_id` уникален в пределах пользователя.

## 9. ИИ-пайплайны

### 9.1. Каталог

Карточка пайплайна содержит:

- название;
- назначение;
- поддерживаемую категорию;
- состояние подключения;
- дату и результат последнего запуска;
- среднюю длительность, если накоплена статистика;
- число результатов, ожидающих проверки;
- кнопку запуска.

В первой версии:

- **Музыка** — активен и подключён к существующему backend-пайплайну;
- **Машинный перевод** — визуальный placeholder «Ещё не подключён»;
- остальные пайплайны не выводятся до появления реализации либо показываются только при наличии записи в серверном реестре.

### 9.2. Единый контракт пайплайна

Интеграция строится через серверный реестр адаптеров. Адаптер обязан уметь:

- проверить входные параметры;
- оценить число элементов;
- создать запуск;
- сообщать прогресс;
- вернуть структурированный результат по каждому элементу и полю;
- вернуть предупреждения и ошибки;
- поддержать отмену, если это возможно;
- повторить только неуспешные элементы;
- применить одобренные изменения идемпотентно.

Интерфейс не зависит от конкретного провайдера модели и не хранит API-ключи.

### 9.2.1. Реальная схема подключения текущего музыкального агента

Существующий агент является CLI, а не HTTP-сервисом:

- entrypoint: `scripts/enrichment-agent/run.mjs`;
- полный цикл: `scripts/music/run-agent-cycle.mjs`;
- адаптер: `scripts/enrichment-agent/adapters/music.mjs`;
- состояния сущностей: `pending/running/completed/review/failed` в `data/enrichment-agent/music/*/state.json`;
- отдельные результаты: `records/*.json`;
- запуски: `runs/*`;
- блокировка параллельного запуска: `run.lock`;
- публикация CLI обновляет staging JSON-файлы, тогда как production runtime сайта читает PostgreSQL.

Для web-интеграции требуется отдельный worker-процесс с concurrency `1` для ключа `music`. API создаёт `pipeline_runs` в PostgreSQL, worker запускает CLI как child process, следит за heartbeat/state-файлом, переносит безопасный структурированный результат в `pipeline_run_items` и завершает DB-run. Директория `data/enrichment-agent` размещается на persistent volume и входит в backup.

Текущий CLI не запускается внутри HTTP-request. Долгий запрос API возвращает `202 Accepted` и `runId`. Прогресс поступает через SSE; разрешён fallback polling раз в 2–5 секунд.

Команда `publish` существующего агента из UI не вызывается. После approval запись преобразуется в runtime music payload и проходит общую рабочую content revision из раздела 7.6.

### 9.3. Запуск музыкального пайплайна

Форма запуска:

1. **Сценарий**
   - «Найти и подготовить новых исполнителей» — `discover`, затем `run` из `run-agent-cycle.mjs`;
   - «Обработать найденных кандидатов» — `run` по `discovered-candidates.json`;
   - «Перепроверить очередь ручной проверки» — `run --retry-review --ai=always`;
   - «Перепроверить выбранные музыкальные карточки» — server формирует временный seed из active item versions и запускает `run` в отдельном scope.
2. **Количество** — `max-items`, по умолчанию 5, допустимо 1–20 в MVP.
3. **AI-режим** — `auto` по умолчанию; `never` доступен только в раскрытом блоке и явно помечен как metadata-only без auto acceptance.
4. **Модель** — фиксированная серверная default `gpt-5-mini`; смена модели доступна только через разрешённый server enum, произвольная строка из браузера запрещена.
5. **Web search** — включён по умолчанию; отключение доступно в расширенных настройках с предупреждением о снижении качества подсказки.
6. **Оценка** — число AI-review calls и ориентировочная стоимость по текущему tariff-config. Если точную стоимость рассчитать нельзя, UI показывает верхнюю оценку и соответствующую пометку.
7. **Подтверждение запуска**.

`OPENAI_API_KEY`, Last.fm/Spotify/TheAudioDB credentials, API base URL, prompt и имя env-переменной никогда не приходят с клиента.

После запуска администратор может покинуть страницу. Задача продолжает выполняться на сервере. В шапке показывается прогресс.

### 9.4. Состояния запуска

- `queued` — ожидает выполнения;
- `running` — выполняется;
- `review_required` — завершён и ожидает проверки;
- `partially_failed` — есть результат и отдельные ошибки;
- `approved` — решения администратора сохранены;
- `staged` — выбранные изменения записаны в content workspace;
- `published` — content revision с этими изменениями активирована;
- `partially_published` — часть выбранных элементов опубликована, часть получила конфликт/ошибку;
- `failed` — запуск не дал проверяемого результата;
- `cancelled` — отменён.

Состояние запуска вычисляется сервером. Перезагрузка страницы не сбрасывает прогресс.

### 9.5. Экран результата

Верхняя сводка:

- обработано / всего;
- успешно;
- с предупреждениями;
- с ошибками;
- без изменений;
- предварительно выбранных к применению;
- длительность и стоимость, если доступны.

Таблица элементов:

- исполнитель/кандидат и связь с существующей карточкой, если она найдена;
- статус;
- число изменённых полей;
- предупреждения;
- уверенность, если pipeline возвращает её;
- чекбокс одобрения;
- кнопка подробного сравнения.

В деталях музыкального результата дополнительно показываются реальные поля агента:

- `assessment`: количество успешных источников, match confidence, наличие обязательных полей и identity conflict;
- `aiReview`: decision, confidence, reasons и использованная модель;
- `hintValidation`: длина, язык, запрещённые совпадения и источники;
- `manualReviewReason`;
- canonical/RU/EN names, artist type, country, city, begin/end year, active state;
- genres, top tracks, top albums, members и popularity metrics;
- source status и кликабельные evidence/source URLs;
- usage токенов и стоимость, если рассчитана.

Сравнение по полям показывает три колонки:

| Поле | Сейчас | Предложено ИИ |
| --- | --- | --- |
| Название поля | текущее значение | новое значение |

Длинный текст сравнивается посимвольно/построчно с подсветкой добавлений и удалений. Изображения показываются рядом. Массивы алиасов показывают добавленные и удалённые элементы.

Для каждого поля можно:

- принять предложение;
- оставить текущее значение;
- вручную отредактировать предложенное значение перед применением;
- отметить проблему в результате.

Массовые действия:

- принять все результаты без предупреждений;
- снять выбор со всех;
- применить решение только к выбранному полю во всех карточках;
- повторить ошибочные элементы;
- экспортировать результат JSON;
- отклонить оставшуюся часть запуска.

Основная кнопка: **«Одобрить и опубликовать выбранное»**. Перед применением показывается итоговое число карточек и полей. Система добавляет изменения в content workspace, строит новую полную revision и просит финальное подтверждение активации. Вторичное действие **«Одобрить в рабочую версию»** оставляет изменения в draft для объединения с другими правками.

### 9.6. Применение

- Предложение преобразуется в runtime payload через функцию, извлечённую из `scripts/music/build-dataset.mjs`; CLI и админка используют одну реализацию mapping.
- Изменения записываются в `content_workspace_changes` транзакционно на уровне одной карточки.
- Для каждой правки сохраняются `source = ai_pipeline`, `run_id`, `run_item_id`, base item version, исходный и итоговый payload.
- Повторный запрос применения с тем же `run_id + item_id` не создаёт дубликат изменений.
- Если карточка изменилась после старта пайплайна, система показывает конфликт и требует ручного сравнения.
- Ошибка одной карточки не откатывает успешно применённые независимые карточки; итог запуска содержит точный отчёт.
- Production меняется только после успешной сборки и активации новой цельной `content_revision`.
- Исходный сырой результат сохраняется для аудита с ограниченным сроком хранения.
- Автоматический `disposition = accepted` агента означает прохождение технических правил. Административное approval остаётся обязательным.

- категория, в которую играл;
- минимальное число сессий.

Поиск: точный/частичный email, user ID, session ID, отображаемое имя.

### 10.2. Карточка пользователя

- данные `user` и `player_profiles`;
- агрегаты активности;
- последние сессии;
- хронология событий;
- отправленные баг-репорты;
- зафиксированные клиентские и серверные ошибки;
- статистика по режимам из `user_mode_stats`;
- attendance/streak;
- wallet balance и последние операции `wallet_ledger`;
- entitlements, promo redemptions и legacy import;
- внутренние заметки администратора;
- история административных действий над пользователем.

### 10.3. Базовое управление

- заблокировать с обязательной причиной;
- разблокировать;
- изменить/сбросить отображаемое имя, если оно пользовательское и нарушает правила;
- добавить внутреннюю заметку;
- скорректировать баланс билетов с обязательной причиной через существующий `/api/v1/admin/wallet-adjustments`;
- завершить все Better Auth sessions пользователя;
- скопировать ID;
- выгрузить данные пользователя в JSON;

Админка не позволяет узнать или установить существующий пароль. Сброс пароля выполняется штатным механизмом авторизации. В первой версии нельзя назначать других администраторов через интерфейс.

Удаление/анонимизация чужого аккаунта из admin UI в MVP не входит. Пользователь уже может удалить собственный аккаунт через текущий `DELETE /api/v1/me`; административная privacy-операция требует отдельной политики хранения финансового ledger, аудита и игровых агрегатов.

При блокировке задаются:

- причина;
- внутренний комментарий;
- срок: бессрочно либо до даты;
- флаг завершения активных сессий.

Для блокировки `player_profiles` расширяется полями `account_status`, `blocked_at`, `blocked_until`, `blocked_reason`, `blocked_by`. `getRequestUser()` после загрузки профиля проверяет статус и возвращает `403 ACCOUNT_BLOCKED` для обычных stateful endpoints. Доступ администратора к собственной панели сохраняет отдельный guard, а self-block запрещён.

## 11. Подробные пользовательские события

### 11.1. Назначение

Журнал должен помогать воспроизвести путь пользователя, понять место сбоя и проверить игровую механику.

### 11.2. Минимальные типы событий

Timeline является read model поверх уже существующих фактов. Игровые действия повторно в отдельную telemetry-таблицу не записываются.

| Событие в UI | Источник |
| --- | --- |
| Аккаунт создан | `user.created_at`, `is_anonymous` |
| Вход/выход/смена доступа | новая `auth_events`; активная session дополняет её текущим состоянием |
| Профиль изменён | `player_profiles.updated_at`, audit для admin-изменений |
| Игра начата | `game_sessions.started_at`, kind/mode/period/difficulty/revision |
| Попытка сделана | `game_attempts.created_at`, position, guessed item version, isCorrect, snapshots |
| Подсказка открыта | `game_hint_choices.created_at`, checkpoint, hintKey, response snapshot |
| Игра завершена | `game_sessions.completed_at`, status, attemptsCount |
| Баг-репорт отправлен | `content_reports.created_at`, reason/comment/status |
| Билеты начислены/списаны | `wallet_ledger.created_at`, type/reason/amount/balanceAfter |
| Период разблокирован | `period_entitlements.unlocked_at` |
| Промокод применён | `promo_redemptions.created_at` |
| Legacy данные импортированы | `legacy_imports.created_at`, counts/warnings |
| Пользователь заблокирован/разблокирован | `audit_log` и поля профиля |

Для отсутствующих в доменной модели сигналов добавляется компактная таблица `client_events` и batch endpoint. Разрешённые типы в MVP:

- `page_view` с route name;
- `mode_opened`;
- `client_error` с sanitized message/stack fingerprint;
- `api_error` с status, error code и request ID;
- `network_offline` / `network_online`;
- `report_form_opened` / `report_submit_failed`.

Произвольные event names и произвольные большие payload сервер отклоняет. Нажатия каждого элемента интерфейса и текст до отправки ответа не собираются.

### 11.3. Поля события

Unified API timeline возвращает нормализованный объект:

- `id` с префиксом источника, например `attempt:<uuid>`;
- `type`;
- `occurredAt` в UTC;
- `userId`;
- `authSessionId`, nullable;
- `gameSessionId`, nullable;
- `itemId` и `itemVersionId`, nullable;
- `mode`, nullable;
- `title` для списка;
- `summary` без секретов;
- `details` с типизированным payload;
- `requestId`, nullable;
- `sourceTable` для диагностики администратора.

`client_events` хранит `id`, `event_name`, `occurred_at`, `user_id`, `auth_session_id`, `game_session_id`, `route`, `app_version`, нормализованные browser/OS/device, `request_id`, `error_code`, `stack_fingerprint` и ограниченный `properties` JSONB.

### 11.4. Интерфейс

Два режима:

- **Поток** — общая таблица событий;
- **Сессия** — последовательная временная шкала одного пользователя/игрового сеанса.

Фильтры:

- период с быстрыми значениями 1 час / 24 часа / 7 дней / 30 дней;
- пользователь или session ID;
- тип события;
- игровой режим;
- card ID;
- game session ID;
- результат;
- устройство;
- correlation ID;
- только ошибки.

Строка разворачивается и показывает форматированный `properties` JSON. Из события доступны переходы в карточку, пользователя, сессию и связанный баг-репорт.

Экспорт текущей отфильтрованной выборки — CSV и JSON, выполняется фоновой задачей.

### 11.5. Приватность и хранение

- Никогда не логировать пароли, access/refresh token, cookies, ключи API, платёжные реквизиты и полные заголовки авторизации.
- IP хранить только при подтверждённой необходимости; предпочтительны маскирование последнего сегмента либо необратимый суточный hash.
- Полный user agent нормализовать в браузер/ОС/тип устройства; сырой вариант хранить ограниченное время при необходимости диагностики.
- Тексты ответов пользователя хранить только в объёме, нужном для проверки механики, и учитывать их в политике конфиденциальности.
- Базовый срок хранения подробных игровых событий — 90 дней. Агрегаты могут храниться дольше. Технические error payload — 30 дней. Конкретные сроки подтверждаются владельцем до релиза.
- Доменные таблицы игр и ledger сохраняют свои текущие продуктовые сроки. Автоматический retention 30 дней применяется к `client_events`; timeline просто перестаёт показывать удалённые client-only события.
- Удаление/анонимизация пользователя должна разрывать прямую связь событий с персональными данными согласно принятой политике проекта.

### 12.12. Использование существующего `audit_log`

## 14. Контроль качества контента

Отдельный раздел автоматических проверок существенно сократит ручной поиск проблем.

### 14.1. Проверки первой версии

- пропущено обязательное поле;
- пустая или слишком короткая подсказка;
- подсказка содержит ответ, название или запрещённый технический текст;
- дублирующийся ID;
- точный или вероятный дубль названия внутри категории;
- совпадающее изображение у разных карточек;
- битый URL изображения;
- некорректный год/дата/числовой диапазон;
- алиас совпадает с другой карточкой и может давать неоднозначный ответ;
- неизвестное значение enum;
- невалидный внешний ID/URL;
- несоответствие текущей JSON Schema.

### 14.2. Интерфейс

- группировка по правилу и категории;
- уровень `critical / warning / info`;
- переход к карточке;
- массовый перезапуск проверки;
- отметка «проверено, допустимо» с комментарием и сроком действия;
- история возникновения и устранения проблемы.

Проверки с тяжёлыми запросами выполняются в фоне. Результат привязывается к item version или workspace change version, чтобы старое предупреждение не отображалось как актуальное после изменения.

## 15. Экономика

Раздел собирает уже существующие административные возможности в понятный интерфейс.

### 15.1. Промокоды

- список с title, состоянием, периодом, reward, per-user/global limit и числом применений;
- создание через существующий `POST /api/v1/admin/promos`;
- отключение и изменение `endsAt` через существующий PATCH;
- сырой код после создания повторно не показывается, поскольку БД хранит HMAC hash;
- UI предупреждает, что потерянный код восстановить нельзя;
- статистика redemption открывает список пользователей без показа credential data.

### 15.2. Билеты

- поиск пользователя и текущий balance/lifetime earned;
- последние операции append-only `wallet_ledger`;
- корректировка через существующий endpoint с обязательной причиной и UUID `Idempotency-Key`;
- предварительный показ нового баланса;
- запрет отрицательного результата остаётся на DB/API уровне;
- ledger никогда не редактируется и не удаляется из админки.

### 15.3. Системная настройка daily salt

Настройка находится в подразделе «Система → Игровые настройки», а не на главной экономической странице. Изменение требует ввода текущего значения, нового значения, причины и подтверждения влияния на будущие загадки. Используется существующий audited endpoint.

## 16. Система и фоновые задачи

- безопасное сообщение об ошибке и request ID;
- повтор задачи только для идемпотентных обработчиков;
- информацию о текущей версии приложения и последнем деплое, если она доступна.

Зависшей считается задача, которая не обновляла heartbeat дольше порога, заданного для её типа. Такие задачи выделяются на обзоре.

Фоновые задачи выполняет новый `apps/worker` либо отдельный entrypoint в `apps/api`, запускаемый отдельным process/container из той же кодовой базы. Очередь хранится в PostgreSQL. Захват задач использует `FOR UPDATE SKIP LOCKED`, heartbeat и уникальный idempotency key. Redis в MVP не добавляется. Worker запускает music CLI child process с ограниченным окружением, рабочей директорией и concurrency `1`.

Новые production-настройки:

- `ADMIN_USER_IDS` — единственный разрешённый UUID;
- текущий `ADMIN_EMAILS=breneize@yandex.ru`;
- `ENRICHMENT_DATA_ROOT` — persistent каталог агента;
- `WORKER_ID`, `WORKER_POLL_INTERVAL_MS`, `WORKER_HEARTBEAT_INTERVAL_MS`;
- `OPENAI_API_KEY` и опциональные Last.fm/Spotify/TheAudioDB credentials доступны worker-контейнеру;
- web-контейнер не получает AI/API credentials;
- все новые переменные добавляются в `.env.example` без значений секретов и валидируются fail-fast.

## 17. Журнал администратора

Фиксируются:

- вход в админку;
- просмотр экспорта персональных данных;
- создание, изменение, скрытие и восстановление карточки;
- массовая операция;
- изменение статуса репорта;
- запуск, одобрение и применение ИИ-пайплайна;
- блокировка/разблокировка пользователя;
- создание экспорта;
- повтор фоновой задачи;
- изменения безопасных настроек.

Фильтры: действие, сущность, ID, период, результат, request ID. Записи доступны только для чтения.

## 18. UX-требования

- Русский язык интерфейса.
- Часовой пояс в UI — `Asia/Almaty`; в БД даты хранятся в UTC.
- Даты показываются полно: `13 июл 2026, 14:35`, по hover — UTC/ISO.
- У каждого длинного процесса есть прогресс, возможность уйти со страницы и итоговое уведомление.
- Пустые состояния объясняют причину и предлагают следующее действие.
- Ошибки сохраняют введённые данные формы.
- Критические действия используют явные названия: «Скрыть 24 карточки», «Заблокировать пользователя».
- Успешные действия подтверждаются коротким уведомлением и сразу отражаются в списке.
- Навигация и фильтры доступны с клавиатуры; интерактивные элементы имеют focus state и подписи для screen reader.
- Цвет не является единственным индикатором состояния.
- Таблицы используют sticky header, закрепляемую первую колонку и настройку видимых колонок.
- Для списков применяются skeleton-состояния без скачков ширины.
- Основные рабочие экраны не открывают каждый объект в новой странице; URL при этом должен содержать ID открытой сущности и поддерживать прямую ссылку.

### 18.1. Базовая компоновка

- постоянный левый sidebar 232–248 px;
- верхняя панель 56–64 px;
- рабочая область использует всю доступную ширину, без маркетингового max-width;
- основной горизонтальный padding 24 px;
- sticky-панель поиска/фильтров над таблицей;
- высота обычной строки 52–56 px, компактный режим 40–44 px;
- detail drawer карточки/репорта 760–960 px и не менее 55% viewport на экранах до 1440 px;
- при ширине меньше 1180 px detail открывается отдельной страницей с сохранением back URL;
- формы используют две колонки только для коротких совместимых полей; подсказки, массивы, источники и diff занимают полную ширину.

Визуальный язык использует брендовые цвета и типографику «Сходится!», сохраняя высокую плотность рабочего интерфейса. Декоративные иллюстрации и крупные игровые карточки на административных списках не используются. Реальная игровая карточка появляется во вкладке предпросмотра.

## 19. Производительность и надёжность

- Все крупные списки используют серверную пагинацию и индексируемые фильтры.
- Цель для загрузки обычной страницы списка — p95 до 1 секунды на сервере при объёме до 50 000 карточек и до 1 000 000 событий за активный срок хранения; итоговый порог уточняется после замера production-БД.
- В браузер не загружается вся база карточек или событий.
- Изменяющие запросы защищены от случайного повтора idempotency key там, где повтор опасен.
- Фоновые задачи имеют heartbeat, число попыток, backoff и dead-letter/failed состояние.
- Музыкальный pipeline переживает перезапуск web-процесса; состояние хранится в БД/устойчивой очереди.
- Применение результата ИИ и восстановление ревизии используют транзакции.
- Для действий с карточкой действует optimistic locking.
- Логи интерфейса не содержат сырой payload с секретами.
- Должен существовать проверенный способ резервного копирования данных, затрагиваемых миграциями; восстановление проверяется до production-релиза.

## 20. Наблюдаемость

Для каждого административного запроса создаётся `request_id`. Для запуска pipeline — `run_id`, для фоновой задачи — `job_id`.

Метрики:

- число и длительность admin API-запросов;
- ошибки по endpoint и коду;
- глубина очереди;
- длительность и успех pipeline runs;
- количество конфликтов версий;
- ошибки применения результатов;
- число новых/закрытых баг-репортов;
- задержка между событием пользователя и его появлением в журнале.

Критические сбои отображаются в админке. Подключение внешних каналов оповещений можно выполнить позднее.

## 21. Критерии приёмки

### 21.1. Доступ

1. `breneize@yandex.ru` входит штатным способом и открывает `/admin`.
2. Проверка разрешения основана на его фактическом user ID.
3. Любой другой пользователь получает отказ от каждого admin API, даже при ручном запросе.
4. Неавторизованный запрос не получает фрагменты административных данных.

### 21.2. Карточки

1. Администратор находит карточку по ID, основному и альтернативному названию.
2. Комбинация фильтров воспроизводится после перезагрузки/по прямой ссылке.
3. Изменение валидного поля сохраняется в server-side workspace и появляется в игровом предпросмотре.
4. Active item version до публикации не меняется.
5. Невалидное поле не попадает в готовую revision; интерфейс показывает конкретную причину.
6. Публикация создаёт полный immutable snapshot всех шести режимов, переводит его в `ready` и активирует только после подтверждения.
7. Архивные и текущие game sessions продолжают ссылаться на прежние item versions.
8. Взятие прошлой версии создаёт workspace change и не переписывает историю.
9. Конфликт base revision/workspace version не перетирает более свежие данные.
10. Массовое действие показывает итог по успешным и ошибочным записям.

### 21.3. Баг-репорты

1. Репорт открывается вместе с карточкой и снимком данных на момент игры.
2. Из репорта можно перейти в редактор, сохранить исправление и связать ревизию.
3. Закрытие требует типа решения и фиксируется в аудите.
4. Дубликат сохраняет ссылку на основной отчёт.

### 21.4. Музыкальный ИИ-пайплайн

1. Запуск принимает допустимую выборку музыкальных карточек.
2. Задача продолжает работу после ухода со страницы.
3. Прогресс восстанавливается после перезагрузки.
4. Результат показывает diff по карточкам и полям.
5. Можно принять отдельные поля, отклонить другие и вручную поправить предложение.
6. Кнопка «Одобрить и опубликовать выбранное» переносит только выбранные поля в workspace и запускает общий publish workflow.
7. Каждое изменение сохраняет run ID и run item ID; созданная content revision содержит итоговые payload.
8. Повтор применения не дублирует изменения.
9. Изменённая после старта pipeline карточка попадает в конфликт и не перезаписывается молча.
10. Ошибочные элементы можно повторить отдельно.

### 21.5. Пользователи и события

1. Пользователь находится по email и ID.
2. Его игровая сессия собирается в последовательную временную шкалу.
3. Из события открываются связанная карточка и баг-репорт.
4. Блокировка требует причины, фиксируется в аудите и реально ограничивает доступ.
5. Секреты и токены отсутствуют в UI, ответах API и экспортируемых событиях.

## 22. Тестирование

Обязательные уровни:

- unit-тесты схем валидации, diff, прав доступа, переходов статусов;
- интеграционные тесты admin API с разрешённым и запрещённым пользователем;
- интеграционный тест адаптера музыкального pipeline на тестовом запуске;
- тест идемпотентности применения;
- тест конфликта версий;
- e2e: поиск и изменение карточки;
- e2e: баг-репорт → исправление → закрытие;
- e2e: pipeline → review → частичное применение;
- e2e: просмотр timeline пользователя;
- e2e: другой пользователь вручную открывает `/admin` и получает отказ от UI и API;
- e2e: draft не влияет на активную игру до публикации;
- e2e: старая игровая сессия продолжает работать после активации новой content revision;
- проверка доступности основных форм и таблиц;
- миграции на копии production-схемы;
- smoke-тест после деплоя.

Тестовые вызовы ИИ не должны использовать production-карточки без отдельной тестовой выборки.

Финальный quality gate использует существующие команды репозитория: `npm run lint`, `npm run typecheck`, `npm test`, `npm run test:integration`, `npm run test:e2e`, `npm run build`, `npm run build:api`, `npm run data:validate`, `npm run data:audit:music`, Docker build и `npm run yandex:bundle`. Yandex-сборка не должна включать доступный admin runtime и должна продолжать собираться.

## 24. Definition of Done

## Ключевое продуктовое решение

Центром админки должен стать единый рабочий экран карточек: список остаётся слева, редактор и связанные репорты открываются рядом, предпросмотр показывает реальную игровую карточку, а сохранение создаёт отслеживаемое изменение в серверной рабочей версии. Публикация собирает новый полный immutable snapshot через уже принятую архитектуру контента. ИИ использует тот же контур: предлагает diff, администратор выбирает поля, система сохраняет происхождение правки и выпускает её общим способом. Такая модель остаётся простой для одного администратора и готова к будущему переводу и другим пайплайнам.
