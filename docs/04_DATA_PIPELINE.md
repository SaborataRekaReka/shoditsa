# Data Pipeline

## Главный принцип

MVP должен работать как статичный сайт. Данные подготавливаются заранее build-time скриптом и сохраняются в JSON.

На клиенте нельзя делать запросы к Кинопоиску/IMDb для каждой игры. Это замедлит сайт, создаст зависимость от лимитов и может раскрыть API-ключи.

## Рекомендуемая схема

1. Получить список популярных фильмов и сериалов.
2. Обогатить каждый тайтл данными: названия, год, жанры, актёры, режиссёры, студии, рейтинги, постеры.
3. Нормализовать поля.
4. Рассчитать `popularityScore`.
5. Отфильтровать мусор и тайтлы с плохими данными.
6. Сохранить `movies.generated.json` и `series.generated.json`.
7. Положить итоговые JSON-файлы в `public/data`.
8. Сайт загружает JSON как статичный asset.

## Источники данных

### Кинопоиск

Пользователь хочет использовать Кинопоиск как главный ориентир для русскоязычной аудитории: топы, русские названия, рейтинг Кинопоиска, популярность в СНГ.

Прямой парсинг HTML Кинопоиска может быть нестабильным и юридически рискованным. Для MVP-пета технически можно делать локальный одноразовый импорт, но лучше заложить слой `provider`, чтобы источник можно было заменить.

Практичные варианты:

1. Использовать неофициальный API kinopoisk.dev или kinopoiskapiunofficial.tech, если условия использования и лимиты подходят.
2. Купить или получить готовый дамп у агрегатора, если проект дойдёт до монетизации.
3. Составить стартовый список ID вручную или полуавтоматически, а детали подтягивать через API.
4. Для рейтингов IMDb использовать официальные IMDb datasets там, где это разрешено условиями.

### IMDb

IMDb предоставляет официальные non-commercial datasets. Они подходят для личного и некоммерческого использования, но условия ограничивают скрейпинг сайта и коммерческое переиспользование данных. Если проект будет монетизироваться, нужно отдельно проверить лицензию или заменить источник на коммерчески разрешённый.

## Размер статичного JSON

500–3000 тайтлов — нормальный объём для статичного сайта, если:

- хранить картинки как URL, а не base64;
- не хранить длинные описания, рецензии и лишние поля;
- грузить отдельно `movies` и `series`;
- использовать gzip/brotli на хостинге;
- индекс поиска строить на клиенте только по нужным полям.

Для MVP можно начать с 500 фильмов и 300 сериалов. Архитектура должна спокойно выдерживать 3000+ тайтлов.

## Что хранить в клиентском JSON

Хранить:

- id;
- mode;
- titleRu;
- titleOriginal;
- alternativeTitles;
- year/endYear;
- countries;
- originalLanguage;
- genres;
- ageRating;
- runtimeMinutes;
- directors/showrunners;
- cast до 5 человек;
- studios;
- ratings;
- votes;
- popularityScore;
- budget;
- posterUrl;
- backdropUrl;
- seasonTags;
- franchise.

Не хранить в MVP:

- полные синопсисы;
- рецензии;
- десятки актёров;
- все изображения;
- трейлеры;
- внутренние сырые ответы API.

## Скрипты

Рекомендуемые скрипты:

```json
{
  "scripts": {
    "data:import": "tsx scripts/import-data.ts",
    "data:normalize": "tsx scripts/normalize-data.ts",
    "data:validate": "tsx scripts/validate-data.ts",
    "data:build": "npm run data:import && npm run data:normalize && npm run data:validate"
  }
}
```

## Структура скриптов

```txt
scripts/
  import-data.ts
  normalize-data.ts
  validate-data.ts
  providers/
    kinopoisk-dev.provider.ts
    imdb-datasets.provider.ts
    manual.provider.ts
  utils/
    normalize-title.ts
    popularity-score.ts
    genre-map.ts
```

## Первая реализация импорта

Для первого рабочего MVP не нужно сразу строить идеальный парсер.

Достаточно:

1. Взять `data/movies.seed.example.json` и `data/series.seed.example.json`.
2. Расширить их до 50–100 тайтлов вручную или через API.
3. Убедиться, что игра работает.
4. Потом автоматизировать импорт 500–3000 тайтлов.

## Валидация данных

Скрипт валидации должен проверять:

- уникальность `id`;
- наличие русского названия;
- наличие года;
- корректность `mode`;
- рейтинг в диапазоне 0–10;
- `popularityScore` в диапазоне 0–100;
- не больше 5 актёров в `cast`;
- не больше 5 жанров;
- отсутствие пустых строк в массивах.

## Рекомендация по легальности

В документации проекта явно разделить:

- MVP/pet usage;
- future commercial usage.

Перед монетизацией проверить условия источников данных и изображений. Если источник запрещает коммерческое переиспользование, заменить его на лицензированный API/дамп или хранить только минимальные ссылочные данные.
