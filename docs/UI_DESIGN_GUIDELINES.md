# Дизайн-гайд платформы «Сходится!»

Статус: канонический документ  
Версия: 1.0  
Дата: 21 июля 2026 года

Этот документ задаёт обязательные правила интерфейса для пользовательской части платформы «Сходится!». Его цель — сохранять ощущение единого продукта при добавлении игр, коммерческих страниц, профиля, клубных механик и служебных состояний.

Гайд описывает уже реализованную систему. Он не предлагает параллельный визуальный язык. Если новый экран требует нового решения, сначала нужно попытаться собрать его из существующих компонентов и токенов.

## 1. Область действия и приоритет источников

Правила распространяются на:

- главную страницу и каталог игр;
- титульные экраны игр;
- игровые сессии;
- архив, статистику и профиль;
- клуб, спецпоказы и коммерческие экраны;
- модальные окна, меню и системные состояния;
- SEO-блоки внутри пользовательских страниц;
- desktop, tablet и mobile версии.

При принятии дизайн-решения использовать следующий порядок источников:

1. Этот документ.
2. Общие токены из `apps/web/src/styles/tokens.css`.
3. Общие компоненты из `apps/web/src/components/app-shell/AppShell.tsx`.
4. Канонические паттерны из `apps/web/src/styles/screens.css`.
5. Общие игровые компоненты `CategoryTicket` и `GameLaunchControls`.
6. Локальные стили конкретной функции — только для композиции и уникальной механики.

`docs/05_UI_UX_BRIEF.md` является историческим брифом и не заменяет этот гайд.

## 2. Основной принцип

Любой новый экран должен выглядеть так, будто он всегда был частью платформы.

Для этого необходимо:

- сохранять общий shell;
- использовать существующие цветовые переменные;
- использовать Manrope для интерфейса и Oswald для display-текста;
- переиспользовать штатные кнопки и селекты;
- сохранять билетную и бумажную метафору;
- придерживаться общей сетки, радиусов и толщины линий;
- не создавать новый визуальный паттерн для уже решённой задачи;
- сохранять доступность и адаптивность наравне с эстетикой.

Уникальность игры выражается содержанием, иллюстрацией, режимным цветом и устройством игрового поля. Header, навигация, основные действия, титульный вход и системные состояния остаются общими.

## 3. Характер бренда

Визуальная формула платформы:

- тёмный клубный зал;
- светлая бумага, билеты, карточки и архивные документы;
- зелёный цвет основного действия;
- янтарный цвет подсказок, наград и специальных акцентов;
- сдержанная ретро-графика;
- тонкая координатная сетка;
- печатная типографика Oswald;
- аккуратные линии вместо тяжёлых декоративных рамок;
- физические детали: перфорация, вырезы, штампы, корешки и бумажная фактура.

Интерфейс должен быть атмосферным, но не театрально перегруженным. Декор не должен конкурировать с условием игры, ответом или главным действием.

## 4. Дизайн-токены

Новые компоненты не должны вводить собственную палитру, радиусы или тени, если эквивалент уже существует.

| `--color-text` | `#f2ecdf` | основной текст на тёмном фоне |
| `--color-muted` | `#aaa395` | вторичный текст на тёмном фоне |
| `--color-muted-2` | `#858b82` | третичный текст на тёмном фоне |
| `--color-paper-muted` | `#6d665a` | вторичный текст на бумаге |

### 4.2. Цвета игровых режимов

| Режим | Переменная | Цветовой характер |
|---|---|---|
| Фильмы | `--mode-movie-brand` | зелёный |
| Сериалы | `--mode-series-brand` | янтарный |
| Аниме | `--mode-anime-brand` | коралловый |
| Игры | `--mode-game-brand` | глубокий синий |
| Диагнозы | `--mode-diagnosis-brand` | циан |
| Города | `--mode-city-brand` | зелёный |
| Музыка | `--mode-music-brand` | сливовый |

Режимный цвет используется как локальный акцент, а не как полная замена темы. Он может окрашивать иконку, билетную полосу, небольшой badge или состояние выбора. Общие кнопки и shell не перекрашиваются произвольно.

### 4.3. Радиусы

| Переменная | Значение | Использование |
|---|---:|---|
| `--radius-sm` | `10px` | небольшие поля, badges, локальные элементы |
| `--radius-md` | `15px` | кнопки, селекты, билеты и стандартные карточки |
| `--radius-lg` | `22px` | hero и крупные секции |
| `--radius-xl` | `28px` | модальные поверхности и редкие крупные панели |

Нельзя вводить радиусы `17px`, `19px`, `23px` и другие случайные значения без геометрической причины. Для вложенной поверхности допустим `calc(var(--radius-*) - 1px)`.

### 4.4. Тени

- `--shadow-tile` — жёсткая тень интерактивной плитки.
- `--shadow-paper` — тень бумажного билета.
- Мягкая длинная тень допустима для модального окна, выпадающего меню или крупной иллюстрации.
- Не смешивать более двух типов теней на одном компоненте.
- Hover основного действия может увеличивать жёсткую тень на 1 px.

### 4.5. Фон приложения

Базовый фон — `--color-bg` с сеткой 24 × 24 px. Hero главной страницы использует `--color-surface`, сетку 32 × 32 px и `images/Vector.svg`.

Если внутренняя hero-секция должна выглядеть как главная, нужно переиспользовать именно этот паттерн, а не создавать похожий градиент.

## 5. Типографика

### 5.1. Семейства

- `Manrope, Arial, sans-serif` — основной интерфейс, абзацы, кнопки и метаданные.
- `Oswald, sans-serif` — названия игр, крупные числа, display-заголовки, билетные надписи и короткие служебные акценты.

Не использовать дополнительные web-шрифты для отдельной игры. Исключение требует обновления этого гайда и общей системы загрузки шрифтов.

### 5.2. Иерархия

| Уровень | Рекомендация |
|---|---|
| Главный display-заголовок | Oswald 500, `48–60px`, line-height `0.95–1.05` |
| Заголовок титульного экрана | Oswald 500, `36–56px` |
| Заголовок карточки | Oswald 500, `20–40px` в зависимости от формата |
| Заголовок модального окна | Oswald 500, `29–31px` |
| Основной текст | Manrope 400–600, обычно `12–14px` |
| UI label | Manrope 700, `10–12px`, uppercase, tracking `.07–.16em` |
| Метаданные | Manrope 500–700, `10–12px` |

### 5.3. Правила

- Верхний регистр использовать для коротких labels, кнопок и display-заголовков.
- Длинный абзац не переводить в uppercase.
- Не делать основной интерфейс мельче 12 px без необходимости высокой плотности.
- Служебный текст 8–10 px допустим только в компактных игровых протоколах и билетных деталях.
- Не использовать одновременно более трёх заметных размеров текста внутри одной карточки.
- Заголовок не должен конкурировать с главным действием.
- Текст на русском должен переноситься естественно; нельзя фиксировать ширину под одну конкретную строку.

## 6. Пространственная система

До появления отдельных spacing-токенов использовать согласованный набор:

`4, 6, 8, 10, 12, 14, 16, 18, 22, 24, 28, 32, 38, 46px`.

Основной ритм строится вокруг 8 px.

Правила:

- между иконкой и текстом: `6–9px`;
- между соседними controls: `7–10px`;
- внутренний padding небольшого control: `8–12px`;
- padding обычной карточки: `14–24px`;
- padding крупной hero-секции: `24–46px`;
- расстояние между крупными секциями: `24–48px`;
- мобильный боковой gutter: `12–14px`;
- desktop боковой gutter shell: `28px`;
- content не должен касаться декоративного выреза, штампа или иллюстрации.

Если появляется значение вне набора, оно должно быть связано с конкретной геометрией: размером иконки, выреза, постера или линией сетки.

## 7. Общий shell

### 7.1. Фон и контейнер

- Минимальная ширина приложения: `320px`.
- Базовая максимальная ширина пользовательского shell: `1440px`.
- На экранах от `1920px` отдельные общие контейнеры могут расширяться до `1680px`.
- Контент всегда центрируется.
- Страница не должна создавать горизонтальный scroll.

### 7.2. Header

Канонический компонент — `AppHeader`.

- Высота: `64px`.
- Положение: sticky, `top: 0`.
- Фон: `--color-bg`.
- Нижняя граница: `--color-line`.
- Desktop использует полный логотип.
- Mobile до `719px` использует только символ бренда.
- Profile control и dropdown переиспользуются без локальных копий.
- Нельзя создавать отдельный header для новой игры.

Допустимо скрыть второстепенные подписи на mobile, но не менять высоту и расположение бренда.

Полноэкранные экраны входа, регистрации и восстановления доступа — единственное пользовательское исключение из обязательного `AppHeader`. Они сохраняют `BrandLogo`, явную ссылку назад к играм, общие токены, `ActionButton` для основного действия и touch-target не меньше `44 × 44px`. Брендовая кнопка внешнего OAuth-провайдера может иметь собственное оформление.

### 7.3. Footer

Канонический компонент — `AppFooter`.

- Footer не дублирует локальную навигацию страницы.
- Юридические ссылки остаются в общей нижней зоне.
- Новая пользовательская страница должна использовать общий footer, если это не полноэкранная активная игровая сессия.

### 7.4. Назад и Esc

Использовать пару:

- `.screen-back` — квадратная кнопка `44 × 44px`;
- `.keycap-hint` — подпись `Esc` на desktop.

Нельзя заменять её случайной круглой стрелкой, текстовой ссылкой или собственной кнопкой. На mobile подсказка `Esc` скрывается.

Игровой shell поддерживает два штатных профиля ширины:

- `standard` — компактные титульники и обычные игровые сессии; навигационная рейка ограничена стандартной шириной;
- `wide` — иммерсивные режимы с широким игровым полотном; навигационная рейка выравнивается по краю этого полотна.

Один режим обязан использовать один и тот же профиль на титульном экране и в игровой сессии. Нельзя размещать узкую навигационную рейку над широким игровым полотном: кнопка назад будет выглядеть оторванной от интерфейса.

## 8. Адаптивность

Канонические пороги общей системы:

| Ширина | Назначение |
|---:|---|
| до `359px` | узкий mobile, только точечные исправления |
| до `719px` | mobile shell |
| от `720px` | tablet/desktop paddings и расширенные сетки |
| от `860px` | три колонки каталога |
| от `980px` | двухколоночный главный hero |
| от `1160px` | четыре колонки каталога |
| от `1920px` | расширенный wide layout |

Feature-level пороги `560px`, `620px`, `768px`, `820px`, `1080px` допустимы для реального перелома конкретной композиции. Нельзя добавлять новый breakpoint только для исправления одного случайного отступа.

Mobile-first требования:

- controls не мельче `44px` по высоте;
- поля ввода используют шрифт не мельче `16px`, чтобы iOS не увеличивал viewport;
- подписи могут скрываться только при сохранении понятного `aria-label`;
- горизонтальные списки должны скроллиться предсказуемо;
- изображения не должны создавать горизонтальное переполнение;
- основное действие доступно без точного наведения;
- контент не должен зависеть от hover.

Контрольные размеры для визуальной приёмки:

- `390 × 844`;
- `768 × 1024`;
- `1024 × 768`;
- `1280 × 800`;
- `1440 × 900`;
- `1680 × 934`.

## 9. Кнопки и действия

Канонический компонент — `ActionButton`.

### 9.1. Варианты

| Вариант | Назначение |
|---|---|
| `primary` | одно главное действие секции или экрана |
| `secondary` | нейтральное равноправное действие |
| `ghost` | третичное, отмена, выход, малоприоритетное действие |
| `hint` | подсказка или специальное янтарное действие |

Базовая геометрия:

- min-height `48px`;
- горизонтальный padding `18px`;
- радиус `--radius-md`;
- gap `9px`;
- текст 12 px, uppercase;
- иконка около 17 px.

Правила:

- На одном уровне не должно быть двух визуально главных кнопок.
- `primary` не использовать для destructive action.
- Disabled-состояние показывает причину рядом или под кнопкой, если она неочевидна.
- Нельзя создавать новую форму кнопки только для одной игры.
- Иконка не должна дублировать смысл соседней иконки или повторяться внутри одного control без причины.
- Текст действия начинается с глагола: «Начать», «Продолжить», «Открыть», «Сдаться».

Собственная кнопка допустима для узкоспециализированной части игрового поля, но её высота, focus, disabled и цветовая семантика должны соответствовать общей системе.

## 10. Селекты, меню и настройки запуска

Канонические компоненты:

- `GameLaunchControls`;
- `GameOptionSelect`;
- `GameOption`;
- `.game-selector`;
- `.period-control`.

Правила:

- Главное действие и настройка запуска образуют единую композицию.
- Не добавлять отдельную карточку «Комната», «Архив» или «Формат», если выбор помещается в штатный option control.
- При нехватке билетов основная кнопка disabled, причина показана непосредственно рядом с ней.
- Меню самостоятельно выбирает открытие вверх или вниз в зависимости от viewport.
- Dropdown не должен выходить за sticky header или нижний край экрана.
- Опции имеют min-height `44px`.
- Выбранное состояние обозначается не только цветом, но и иконкой/рамкой.

## 11. Поверхности и карточки

### 11.1. Тёмная поверхность

Используется для:

- shell;
- hero;
- навигации;
- вторичных actions;
- фоновых панелей активной игры.

Сочетание: `--color-surface`, `--color-text`, `--color-muted`, `--color-line`.

### 11.2. Бумажная поверхность

Используется для:

- билетов;
- протоколов;
- модальных окон;
- архивных документов;
- результатов и подробных данных.

Сочетание: `--color-paper`, `--color-ink`, `--color-paper-muted`, `--color-paper-line`.

Бумажная текстура берётся из существующих assets. Она должна быть слабой и не снижать контраст текста. Не использовать чисто белый фон для бумажной карточки.

### 11.3. Границы

- Базовая толщина — `1px`.
- На тёмном фоне использовать `--color-line` или `--color-line-strong`.
- На бумаге использовать `--color-paper-line`.
- Dashed-линия допустима для перфорации, отрывного корешка или архивного разделителя.
- Не использовать тяжёлую рамку одновременно с тяжёлой тенью.

## 12. Билетная метафора

Билет — ключевой визуальный паттерн платформы. Канонические реализации:

- `.admit-ticket`;
- `.concert-ticket`;
- `CategoryTicket`;
- клубные тарифные карточки;
- протокол расследования «Данетки».

Билет обычно содержит:

- корешок или визуально отделённую служебную часть;
- основную бумажную часть;
- тонкую внешнюю линию;
- штамп, перфорацию, barcode или небольшой служебный знак;
- один режимный цвет;
- один главный интерактивный маршрут.

Не нужно помещать на билет все декоративные элементы сразу.

### 12.1. Вырезы

Вырез — это физическое отсутствие материала карточки. Внешняя граница карточки не должна проходить поперёк открытой стороны выреза.

Обязательные правила:

- сохранять дугу внутреннего края выреза;
- у левого выреза убирать `border-left`, у правого — `border-right`;
- у верхнего выреза убирать `border-top`, у нижнего — `border-bottom`;
- фон выреза должен совпадать с фоном непосредственно за карточкой;
- элемент выреза должен перекрывать внешнюю границу карточки, включая однопиксельную линию;
- если `overflow: hidden` не позволяет перекрыть границу, нужно вынести клиппинг во внутренний wrapper или использовать mask;
- оставлять линию через открытие выреза запрещено;
- внутренний divider также не должен проходить через вырез;
- правило действует для билетов, тарифов, протоколов, купонов и любых новых фигурных карточек.

Канонический паттерн боковых вырезов:

```css
.ticket {
  position: relative;
  overflow: visible;
  border: 1px solid var(--color-paper-line);
}

.ticket::before,
.ticket::after {
  content: '';
  position: absolute;
  width: 20px;
  height: 40px;
  border: 1px solid var(--color-paper-line);
  background: var(--color-bg);
  box-sizing: border-box;
  pointer-events: none;
}

.ticket::before {
  left: -1px;
  border-left: 0;
  border-radius: 0 20px 20px 0;
}

.ticket::after {
  right: -1px;
  border-right: 0;
  border-radius: 20px 0 0 20px;
}
```

Перед выпуском вырезы проверяются на тёмном и светлом фоне при масштабе 100% и 125%.

## 13. Главная страница

### 13.1. Hero

Канонический `.hub-hero`:

- поверхность `--color-surface`;
- border `--color-line`;
- radius `--radius-lg`;
- сетка 32 × 32 px;
- фоновый `Vector.svg`;
- одна copy-колонка и одна visual-колонка на desktop;
- высота около 320 px на широком desktop;
- главный заголовок 50–60 px;
- иллюстрация не обрезает смысловые детали.

Новый hero не должен вводить иной shell, отдельный логотип или несовместимую CTA-кнопку.

### 13.2. Карточки игр

Использовать `CategoryTicket` и конфигурацию `CATEGORY_TICKET_CONFIG`.

- Режимный цвет поступает через конфигурацию.
- Статусы: `new`, `active`, `completed`.
- Карточка должна быть ссылкой, если имеет самостоятельный URL.
- Вся карточка является одной интерактивной областью.
- Нельзя дублировать действие отдельной кнопкой внутри кликабельной карточки.
- Название, описание, состояние и действие имеют фиксированную иерархию.

## 14. Титульные экраны игр

Канонический контейнер — `.title-screen`.

Структура:

1. Общий `AppHeader`.
2. Стандартная строка «Назад» + `Esc`.
3. `.title-stage` с режимной иконкой и названием.
4. Дата и короткое описание.
5. Билетная или тематическая карточка игры.
6. `GameLaunchControls`.
7. Одно главное действие: «Начать», «Продолжить» или «Результат».
8. SEO-разворот/описание ниже основного входа, если он требуется странице.

Размеры:

- контейнер до `960px`;
- основная сцена до `640px`, на ultra-wide до `720px`;
- верхний padding около `46–58px`;
- игровая иконка `48 × 48px`;
- play button min-height `58px`.

Запрещено:

- делать отдельный header;
- заменять стандартную кнопку запуска уникальной кнопкой;
- добавлять дублирующие controls «Комната», «Архив», «Формат»;
- скрывать стоимость до момента клика;
- переносить важное SEO-описание только внутрь недоступного изображения.

## 15. Игровые сессии

Базовый контейнер `.game-shell` имеет ширину до `820px`, на ultra-wide — до `960px`. Более широкий layout допустим для механики, объективно требующей пространства: совместная доска, протокол или многоколонная сцена.

Даже широкий игровой модуль обязан сохранять:

- `AppHeader`;
- общий фон приложения;
- стандартную кнопку назад;
- общие button variants;
- общие tokens;
- одинаковые состояния loading/error/offline;
- отсутствие горизонтального scroll.

### 15.1. Иерархия сессии

1. Навигация назад.
2. Название/метаданные текущей игры.
3. Главное игровое содержание.
4. Вторичная информация или история.
5. Поле ввода/управление.
6. Нижние действия.

Не переносить главное действие в декоративную область. Не создавать два независимых набора нижних действий.

### 15.2. Совместная игра

- Участники показываются компактными аватарами.
- После двух-трёх видимых аватаров используется `+N`.
- Число участников имеет правильное склонение.
- Пользовательские сообщения расположены справа, ведущий/система — слева.
- История едина и хронологична; две независимые колонки истории запрещены.
- Статусы realtime не должны перекрывать контент.

### 15.3. «Данетка» как wide-layout

«Данетка» использует широкий протокол, но остаётся частью общего shell.

- Hero использует паттерн главной hero-секции.
- Иллюстрация размещается отдельным прозрачным asset через `object-fit: contain`.
- Важные части персонажа не обрезаются.
- Номер дела формируется из данных, а не зашивается в raster.
- Статус ведущего находится в отдельной строке.
- Протокол использует бумажную поверхность и билетные вырезы.
- Чат остаётся компактным и хронологичным.

## 16. Формы и ввод

- Label всегда связан с input семантически или через понятный `aria-label`.
- Высота поля — не менее `44px`.
- На mobile размер шрифта — не менее `16px`.
- Placeholder не заменяет label, если назначение поля неоднозначно.
- Focus виден на тёмном и светлом фоне.
- Ошибка располагается рядом с полем и содержит понятное действие.
- Поле не должно менять ширину или прыгать при появлении ошибки.
- Кнопка отправки disabled при пустом или невалидном значении.

## 17. Состояния

Каждый интерактивный модуль должен иметь:

- default;
- hover для устройств с курсором;
- active;
- focus-visible;
- disabled;
- loading;
- empty;
- error;
- success/completed, если применимо;
- offline/reconnecting для realtime-функции.

### 17.1. Loading

- Сохраняет ожидаемую геометрию экрана.
- Не показывает бесконечный spinner без пояснения, если операция длительная.
- Не блокирует весь shell без необходимости.

### 17.2. Empty

- Объясняет, почему данных нет.
- Предлагает одно релевантное следующее действие.
- Не выглядит как ошибка.

### 17.3. Error

- Сообщение написано человеческим языком.
- Технический код не является главным текстом.
- Retry показывается там, где повтор действительно возможен.
- Ошибка не перекрывает бренд и основную навигацию.

### 17.4. Disabled

- Сохраняет читаемость.
- Не реагирует на hover.
- Если причина неочевидна, рядом показывается объяснение: стоимость, нехватка билетов, завершённая игра или отсутствие данных.

## 18. Модальные окна и dropdown

Канонический `Modal` использует focus trap и возвращает фокус вызывающему control.

- Mobile: bottom sheet с верхними радиусами.
- Desktop от `720px`: центрированная карточка с полным радиусом.
- Backdrop: тёмный, контрастный, но не абсолютно чёрный.
- Максимальная ширина стандартной modal: около `540px`.
- Закрытие: кнопка, Escape, допустимый клик по backdrop.
- В modal должна быть одна визуально главная кнопка.
- Dropdown ограничивается viewport и имеет собственный scroll.

## 19. Иконки и иллюстрации

### 19.1. Иконки

- Основная библиотека — `lucide-react`.
- Обычная UI-иконка: `14–18px`.
- Stroke width около `1.8–2`.
- Иконка дополняет текст, а не заменяет непонятное действие.
- Не смешивать filled, outline и emoji-иконки в одном control group.
- Emoji допустимы только как содержательный элемент игры, не как системная иконка.

### 19.2. Бренд

- Desktop: `images/logo.svg`.
- Mobile: `images/symbol.svg`.
- Нельзя перерисовывать знак локально или собирать его из emoji.

### 19.3. Иллюстрации

- Прозрачный персонаж размещается отдельным `<img>`/`picture`.
- Для персонажей по умолчанию использовать `object-fit: contain`.
- `cover` допустим для фото/poster, где осознанная обрезка входит в композицию.
- Голова, лицо, руки и предмет взаимодействия не обрезаются без художественной причины.
- Не вшивать динамический текст, дату, цену или номер дела в raster.
- Добавлять `width` и `height`, чтобы избежать layout shift.
- WebP/AVIF использовать вместе с безопасным fallback, если это необходимо.

## 20. Движение

Каноническая easing-функция: `--ease`.

Разрешено:

- появление экрана снизу на `8–14px`;
- лёгкое смещение кнопки на hover;
- fade для backdrop;
- вращение spinner;
- раскрытие menu/sheet;
- короткая status-анимация.

Запрещено:

- постоянное мигание;
- тяжёлый parallax;
- bounce у каждого элемента;
- анимация, мешающая чтению;
- transition layout-свойств, вызывающий заметные скачки.

При `prefers-reduced-motion: reduce` анимации и transitions отключаются.

## 21. Доступность

Обязательный минимум:

- все интерактивные элементы доступны с клавиатуры;
- focus-visible не удаляется;
- контраст соответствует назначению текста;
- смысл не передаётся только цветом;
- icon-only control имеет `aria-label` и `title`, если это полезно;
- декоративное изображение имеет пустой `alt` и `aria-hidden`;
- информативное изображение имеет содержательный `alt`;
- modal имеет `role="dialog"`, `aria-modal` и управляемый фокус;
- realtime-лог использует подходящий `aria-live`/`role="log"` без спама;
- touch target не меньше `44 × 44px`;
- текст остаётся читаемым при zoom 200%;
- страница работает без hover.

## 22. Тексты и локализация

- Основной язык интерфейса — русский.
- Использовать «ё» там, где она улучшает читаемость.
- Склонять числа: `1 участник`, `2 участника`, `5 участников`.
- Не использовать транслит в пользовательском интерфейсе.
- Не показывать внутренние идентификаторы вместо названий.
- Дата и время имеют единый локальный формат внутри одного экрана.
- Labels короткие; подробное объяснение выносится в description.
- Состояния подключения формулируются понятно: «На связи», «Восстанавливаем связь», «Не в сети».

## 23. Архитектура UI-кода

Перед созданием нового компонента проверить:

1. Можно ли использовать `ActionButton`?
2. Можно ли использовать `AppHeader`, `AppFooter`, `Modal`?
3. Можно ли собрать выбор через `GameOptionSelect`?
4. Можно ли представить игру через `CategoryTicket`?
5. Существует ли похожая поверхность в `screens.css`?
6. Существует ли нужный token в `tokens.css`?

Если один новый паттерн нужен на двух и более экранах, он должен стать общим компонентом или общим token. Копирование CSS между feature-файлами запрещено.

Локальный feature CSS может:

- задавать уникальную сетку игрового поля;
- размещать специфическую иллюстрацию;
- оформлять уникальные игровые данные;
- адаптировать общие компоненты в рамках их контракта.

Локальный feature CSS не должен:

- создавать новый header/footer;
- переопределять глобальную палитру;
- дублировать `.ui-button`;
- вводить несовместимые focus/disabled-состояния;
- ломать общий scrollbar;
- использовать `!important` как обычный способ композиции;
- зависеть от конкретного пользовательского текста.

## 24. Решение о новом паттерне

Новый визуальный паттерн допустим только когда:

- задача отсутствует в общей системе;
- существующий компонент нельзя расширить без ухудшения других экранов;
- паттерн описывает повторяемый продуктовый сценарий;
- определены состояния, адаптивность и доступность;
- обновлён этот документ;
- при необходимости создан общий компонент и token.

Если решение нужно только для одной картинки или одного отступа, новый паттерн не создаётся.

## 25. Процесс дизайн-изменения

1. Найти ближайший реализованный аналог.
2. Проверить общие tokens и компоненты.
3. Зафиксировать, что остаётся неизменным в shell.
4. Реализовать композицию mobile-first.
5. Проверить пустое, заполненное, loading и error-состояния.
6. Проверить keyboard и focus-visible.
7. Проверить контрольные viewport.
8. Сравнить с соседними экранами платформы.
9. Проверить отсутствие горизонтального scroll.
10. При добавлении нового общего правила обновить этот гайд.

## 26. Визуальная приёмка

Перед выпуском ответить «да» на каждый вопрос.

### Shell

- Использован общий header?
- Использована стандартная кнопка назад?
- Фон совпадает с соседними экранами?
- Контейнер соответствует общей ширине?
- Нет горизонтального scroll?

### Компоненты

- Использованы штатные кнопки?
- Главное действие ровно одно?
- Disabled-состояние объяснимо?
- Нет дублирующих controls или иконок?
- Dropdown и modal помещаются во viewport?

### Типографика

- Display-текст использует Oswald?
- Основной текст использует Manrope?
- Иерархия считывается без цвета?
- Нет случайных размеров и межбуквенных интервалов?
- Длинный русский текст переносится корректно?

### Билеты и бумага

- Использованы бумажные tokens?
- Текстура достаточно слабая?
- Вырезы действительно разрывают внешнюю границу?
- Divider не проходит через вырез?
- Корешок и основная часть не спорят между собой?

### Изображения

- Важные части иллюстрации не обрезаны?
- Динамические данные не зашиты в raster?
- Указаны размеры изображения?
- Есть корректный alt или пустой alt для декора?

### Состояния и доступность

- Проверены loading, empty, error и disabled?
- Focus-visible видим?
- Touch targets не меньше 44 px?
- Интерфейс понятен без hover?
- `prefers-reduced-motion` соблюдается?

### Responsive

- Проверены 390, 768, 1024, 1280, 1440 и 1680 px?
- На mobile не пропало главное действие?
- На коротком desktop нет наложений?
- Иллюстрация не перекрывает текст и дату?
- Вложенные scroll-зоны работают предсказуемо?

## 27. Краткий список запретов

Нельзя:

- создавать отдельный визуальный язык для новой игры;
- копировать header или footer;
- рисовать новую основную кнопку вместо `ActionButton`;
- использовать чисто белые бумажные карточки;
- проводить границу через билетный вырез;
- дублировать иконки в одном control;
- вшивать цену, дату или номер в иллюстрацию;
- скрывать причину disabled-действия;
- уменьшать touch target ниже 44 px;
- полагаться только на цвет;
- добавлять breakpoint для единичного случайного дефекта;
- выпускать экран без mobile-проверки;
- изменять общий shell ради локальной композиции.

## 28. Definition of Done для нового экрана

Экран считается готовым, когда:

- он собран из общей системы;
- его уникальные элементы не дублируют общие компоненты;
- он визуально совместим с главной, титульными и соседними игровыми экранами;
- все обязательные состояния реализованы;
- desktop и mobile версии не имеют переполнения и наложений;
- доступность проверена клавиатурой;
- production build проходит;
- общие новые правила внесены в этот документ.
